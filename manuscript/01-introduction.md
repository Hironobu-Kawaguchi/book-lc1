# 序論：なぜ GitHub と Markdown か

従来の WYSIWYG エディタは差分が読みにくくレビューが困難である。
Git はテキスト差分に最適化されており Markdown と相性が良い。
Docs-as-Code の採用により執筆はソフトウェア開発と同様に管理できる。
Pandoc を使えば Markdown から高品質な EPUB3 を直接生成できる。

## 1 文 1 行の原則

GitHub のレビューを最大限に活かすために 1 文 1 行で書く。
この規約は diff をノイズレスにしコメント位置を正確にする。

## リポジトリ構成の意図

章をファイル単位に分割すると並行作業とマージが容易になる。
`metadata.yml` は書籍全体の設定を集中管理する。
`epub.css` は見た目の一貫性と読書体験の品質を担保する。

## 画像と表紙

画像は `images/` に置き相対パスで本文から参照する。
大きな画像は Git LFS で追跡してリポジトリの肥大化を防ぐ。
表紙画像は `metadata.yml` の `cover-image` で指定する。
